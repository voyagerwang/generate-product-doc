# 输入目录约定（建议）

## 最小可用

- `screenshots/`：页面截图（按页面命名）
- `scope.md`：本次覆盖页面范围

## 推荐

- `examples/`：历史高质量文档（建议3-10篇）
- `rules.md`：硬规则（优先级最高）
- `glossary.md`：术语/缩略词定义
- `prototype/`：原型源码或页面链接
- `flows/`：已有流程图素材（可选）

## rules.md 建议字段

### 1) 文档结构
- 是否完整PRD
- 是否固定`4.2 功能清单`
- 是否保留/删除总结章节
- 是否需要单列“需求范围说明”（默认不单列，避免与功能清单重复）

### 2) 表格结构
- 功能清单表头（建议：平台/一级功能/二级功能/优先级/说明）
- 功能详设表头（建议：功能模块/元素/逻辑说明）
- 是否每页单独表格
- 列宽比例（如20/40/40）

### 3) 元素写法
- 字段句式：`字段名，字段类型，字段要求`
- 字段顺序：从上到下、从左到右
- 列表模块是否固定写：展示范围/展示顺序/字段说明/字段交互

### 4) 逻辑写法
- 强制覆盖：入参校验、状态流转、计算口径、异常提示
- 复杂字段是否要求阈值/并列/边界规则
- 禁止纯技术实现细节

### 5) 流程图策略
- 哪些模块必须补流程图
- 流程图放置位置（截图后或模块后）
- 渲染优先级（飞书组件 > 图片 > 文本兜底）

### 6) 交付策略
- 目标文档（飞书链接或本地路径）
- 未提供飞书时的默认格式（建议：`docx`）
- 更新方式（局部更新优先；仅用户明确要求时整文重写）
- 权限要求（是否“链接可编辑”）

## 默认交付行为

1. 提供飞书链接/ID：写入指定飞书文档。
2. 未提供飞书链接/ID：默认写本地 `docx`。
3. 仅当 `docx` 生成能力不可用时：降级为本地 `md`，并返回降级说明。
