# 产品文档生成（公开版）

用于“先抽取风格，再按业务事实落地 PRD”的通用文档生成。

## 默认执行模式（必须）

当用户只说“写文档/生成PRD/按目录出文档”时，默认执行：
1. 读取目录中的样本、截图、原型、规则。
2. 建立风格画像（结构、术语、序号、颗粒度）。
3. 先确认输出目标是“飞书文档”还是“本地 `docx`”。
4. 先抽页面骨架，再写正文；不直接从源码和原型逐段转写。
5. 输出前必须执行自检，不合格页面重写。

若用户选择飞书：
1. 必须先拿到飞书文档链接或 `docId`。
2. 必须检查 `FEISHU_APP_ID` 与 `FEISHU_APP_SECRET` 是否已配置。
3. 缺任一项时先提示补充，不自动降级。

若用户选择本地：
1. 默认输出 `docx`。
2. 仅在无法生成 `docx` 时回退 `Markdown`。

## 输入约定

优先读取（存在即用）：
- `examples/`：历史高质量文档
- `screenshots/`：页面截图
- `prototype/`：原型源码或页面入口
- `rules.md`：硬规则（优先级最高）
- `glossary.md`：术语表
- `scope.md`：页面范围

详细参考：`references/input-contract.md`

## 输出结构（默认模板）

1. 文档信息（表格）
2. 项目背景与目标
3. 用户场景与用户故事
4. 信息架构与功能清单
4.1 信息架构（代码块树形）
4.2 功能清单（表格）
5. 功能详细说明
6. 非功能性需求

## 4.2 功能清单（固定表头）

默认使用表格：
- `平台 | 一级功能 | 二级功能 | 优先级 | 说明`

硬规则：
1. 只写模块级能力，不写页面内原子操作。
2. 二级功能优先对应页面、页签或独立业务模块。
3. 上架、下架、删除、导出、下载、设为默认、筛选、重置等动作写入 `5` 章，不在 `4.2` 单独占行。
4. 说明列写“业务能力边界 + 核心约束”，不写实现细节。

## 5 章写法（核心约束）

### 标题层级

第 `5` 章标题树必须与 `4.1 信息架构` 保持一致，不允许把所有页面平铺在同一层级。

### 页面组织

每个正式页面按以下顺序输出：
1. 页面标题
2. 原型截图
3. 功能详设表

若缺少截图：
1. 优先从原型补抓截图。
2. 无法补抓时，该页面只能标记为“草稿版页面说明”。
3. 不允许写“本次未提供独立截图，页面以源码原型为准”这类正式版伪定稿表述。

### 页面模板

#### 列表页

默认顺序：
1. 信息汇总
2. 主按钮
3. 筛选区
4. 列表
5. 分页
6. 空状态

#### 表单页

默认顺序：
1. 表单分组
2. 字段定义
3. 提交按钮
4. 取消/返回
5. 提交校验
6. 保存成功与失败结果

#### 详情页

默认顺序：
1. 概览信息
2. 页签或二级导航
3. 当前页签主信息
4. 子模块
5. 操作区

#### 占位页

默认不为每个占位导航页单独展开完整章节。
1. 默认合并成“占位模块汇总说明”。
2. 仅当用户明确要求逐页展开时，才为每个占位页独立成节。

### 功能详设表默认列

每页独立表时默认三列：
- `功能模块 | 元素 | 逻辑说明`

### 元素列写法

元素按“从上到下、从左到右”描述，但必须先模块后字段，不允许一上来逐字段平铺。

句式：
- `字段名，字段类型，字段要求`

列表/详情/表单模块优先使用以下结构：
1. 展示范围
2. 展示顺序
3. 字段说明
4. 字段交互

通用低价值元素处理规则：
1. 面包屑、页面标题、返回按钮等通用元素，若无业务规则，可压缩描述或省略。
2. 优先写对业务有影响的模块：筛选区、列表、状态、主按钮、提交按钮、关键弹窗。

### 字段定义规则

1. 若原型或代码中有明确枚举值，必须直接展开，不写“取自字典”。
2. 若原型或代码中没有明确枚举值，不要假装写清楚，应列入“待补充信息”。
3. 若字段支持搜索、联想、模糊匹配、精确匹配，必须写清匹配方式。
4. 若字段影响后续流程、其他页面展示或状态流转，必须补影响范围。

### 逻辑说明列写法

写“可验收业务逻辑”，重点覆盖：
1. 入参校验
2. 命中规则/过滤规则
3. 状态流转与触发条件
4. 计算口径与边界处理
5. 异常提示与空态
6. 权限或操作限制（有则写）

不要写：
1. 技术实现细节，例如“停止复用请求、避免重复拉取”。
2. 无效目的句，例如“用于标识当前页面位置”“用于识别课程主体”。

### 技术信息隔离规则

以下内容默认不写入正文功能说明：
1. `index.html`、路由配置、Mock 数据、组件路径等技术来源。
2. URL 参数、前端状态管理实现、接口调用方式。

若这些内容直接影响用户可见行为，才可转译成业务规则写入正文。

## 复杂字段深写规则

命中以下任一条件，必须进入深写模式：
1. 状态字段
2. 时间字段
3. 数量、比例、进度字段
4. 开关或单选后会影响后续流程的字段
5. 会影响其他页面展示的字段
6. 有并列规则、阈值、边界值的字段

深写至少覆盖：
1. 默认值
2. 校验规则
3. 计算口径
4. 边界处理
5. 并列规则（有则写）
6. 影响范围

## 流程图策略

以下场景必须补流程图：
1. 多分支判断
2. 阻断校验
3. 关键状态流转

优先级：
1. 飞书可编辑流程图组件
2. 图片流程图
3. 文本兜底

## 缺失信息处理

若以下信息缺失，不要假装写清楚：
1. 截图
2. 枚举值
3. 状态取值
4. 权限范围
5. 统计口径

处理方式：
1. 能从原型或代码补齐就补齐。
2. 不能补齐则列入“待补充信息”。
3. 缺失会影响定稿质量时，明确标记该页面为草稿版。

## 飞书排版规范

1. 序号优先使用原生有序列表。
2. 一级序号：`1.2.3`；二级序号：`a.b.c`。
3. 同级要点之间不插空行。
4. 信息架构使用代码块呈现树形结构。
5. 表格优先，文本次之；文档信息也使用表格。

## 更新策略

默认：局部更新，不整篇重写。

仅在以下情况整文重写：
1. 用户明确要求“重写整篇/全量覆盖”。
2. 文档结构已严重偏离模板，局部修复成本明显高于重写。

## 质量闸门（生成后必须自检）

若出现以下任一情况，必须重写对应章节：
1. `4.2` 中把原子操作拆成独立二级功能。
2. `5` 章里按字段逐条平铺，没有先按模块组织。
3. 出现“用于标识/用于识别/用于展示/用于确认”这类空话。
4. 枚举字段只写“取自字典”，没有展开值，也没有列入待补充项。
5. 缺少截图却仍以正式定稿语气输出。
6. 占位页写得过重，冲淡主链路。
7. 正文里混入技术来源和实现细节。

## 禁止写法

- `A或B`
- `待研发确认`
- `待定`
- `可选`
- `研发自行决定`
- `取自字典`（未展开枚举时）
- `本次未提供独立截图，页面以源码原型为准`
- 无口径字段
- 泛化空话，如“系统处理后返回结果”
